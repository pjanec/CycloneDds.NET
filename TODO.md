**Specification: Tiered Data Formatting Pipeline**

**1. Architecture Overview**
The data presentation layer for CycloneDDS.NET utilizes a highly flexible, UI-agnostic Tiered Formatting Pipeline to render DDS sample data. This pipeline separates the underlying structural data from its visual representation, enabling native text logging, zero-allocation string interpolation, and rich syntax highlighting in tools like the `DetailPanel`. The system processes values through three cascading tiers of resolution to determine the optimal formatting strategy.

**2. Token-Based Semantic Formatting**
To maintain a strict separation of concerns, formatters do not emit UI-specific markup (like HTML or Blazor `RenderFragment`s). Instead, they yield a sequence of **`FormattedToken`** structures. 
*   Each token consists of a plain text string and a semantic **`TokenType`** (e.g., `String`, `Number`, `Boolean`, `Enum`, `Null`, `Punctuation`, `Keyword`). 
*   The consuming UI (such as Blazor, WPF, or a console app) maps these semantic tokens to its own presentation layer. In the DDS Monitor, the UI maps these tokens to specific CSS classes (e.g., `is-number` or `is-string`).

**3. The Three Resolution Tiers**
When rendering a value, the pipeline attempts to resolve the formatting in the following order:

*   **Tier 1: Custom Interface-Based Formatters (Registry)**
    *   **Mechanism:** Uses a dependency-injected `IValueFormatterRegistry` to discover formatters implementing `IValueFormatter`.
    *   **Behavior:** Formatters evaluate the `Type`, `FieldMetadata`, and current payload, returning a suitability score and providing both `FormatText()` and `FormatTokens()` outputs.
    *   **Application:** If the registry yields a formatter with a high enough match score (e.g., `>= 0.8f`), it is **auto-applied** by default. Otherwise, formatters with lower scores are registered as alternative views that the user can manually select.

*   **Tier 2: Duck-Typed Schema Formatters (CodeGen Defaults)**
    *   **Mechanism:** Driven by schema attributes like `[DdsTypeFormat]`, processed entirely at compile time by the `CycloneDDS.CodeGen` tool.
    *   **Behavior:** The code generator emits a highly optimized, zero-allocation `ToString()` override using native C# string interpolation directly into the generated struct. To support syntax highlighting without forcing a dependency on the `CycloneDDS.Runtime` formatting interfaces, the generator also emits an optional, duck-typed `IEnumerable<FormattedToken> GetFormatTokens()` method.
    *   **Application:** The UI uses reflection to check if the `GetFormatTokens` method exists on the target type. If found, it invokes the method to render the beautifully colored inline preview automatically. This tier provides zero-dependency, auto-generated syntax highlighting out of the box.

*   **Tier 3: Standard Fallback Rendering**
    *   **Mechanism:** The default Blazor rendering logic built directly into the UI components.
    *   **Behavior:** If a value falls through Tiers 1 and 2, it is evaluated by its raw CLR type. 
    *   **Application:** Standard primitives (numbers, booleans), enums, and strings are rendered using fallback logic with basic CSS type mapping. Arrays and fixed-size buffers (`public unsafe fixed T Name[N]`) are rendered compactly, showing only their length (e.g., ``).

**4. UI Integration & Context Menus**
The formatting pipeline integrates seamlessly with the user interface via mouse interactions:
*   **Context Menu:** Right-clicking a value in the `DetailPanel` opens a context menu that dynamically lists all available formatters discovered in Tier 1. 
*   **Default Selection:** The menu includes a "Default" option, which naturally routes the renderer to fall through Tier 1 and rely on the auto-generated Tier 2 duck-typed schema formatter, or the Tier 3 fallback.
*   **Clipboard Support:** All tiers support a plain-text output strategy (either via concatenating the text of tokens, `FormatText()`, or `ToString()`) ensuring the "Copy to Clipboard" action always captures the exact formatted representation.


**Specification Addendum: Attribute-Based Schema Formatting (Tier 2)**

**1. Overview & Philosophy**
The attribute-based formatting system enables developers to define rich, UI-agnostic data presentation rules directly within their C# DDS schema definitions. By utilizing a single class-level attribute with an interpolation-style template, the schema becomes the absolute single source of truth for both network serialization and UI presentation. 

Because the `CycloneDDS.CodeGen` tool parses these templates at compile time, the system guarantees:
*   **Zero Runtime Reflection:** All formatting rules are hardcoded into the generated partial classes.
*   **Native C# Performance:** Text formatting leverages heavily-optimized standard C# string interpolation.
*   **Zero Boilerplate:** Developers do not need to write or manually register separate `IValueFormatter` classes.

**2. The `[DdsTypeFormat]` Attribute**
Formatting is controlled via the `[DdsTypeFormat]` attribute, which is applied to any `[DdsStruct]`, `[DdsTopic]`, or `[DdsUnion]` definition. It accepts a single `Template` string parameter.

**Syntax:**
The template string combines literal text with placeholder expressions enclosed in braces `{}`. 
Each placeholder follows the syntax: `{FieldName:FormatString:TokenType}`

*   **`FieldName`** (Required): The exact name of the C# field or property on the struct.
*   **`FormatString`** (Optional): A standard .NET format string passed directly to `ToString()` (e.g., `D`, `X8`, `0.00`).
*   **`TokenType`** (Optional): The semantic semantic token type used for UI syntax highlighting (e.g., `Number`, `Keyword`, `String`, `Boolean`).

*Literal Text:* Any characters outside of the `{}` braces are preserved exactly as written and are automatically assigned the `TokenType.Punctuation` (or `Default`) classification by the code generator.

**3. Code Generator Behavior (`SerializerEmitter`)**
During the build process, the `SerializerEmitter` scans the syntax tree for the `[DdsTypeFormat]` attribute. When detected, it parses the template using a regular expression and emits two methods directly into the type's generated partial struct:

1.  **`public override string ToString()`**
    *   The code generator translates the custom template directly into a native C# string interpolation expression (e.g., `return $"{Site:D}:{App:D}:{Entity:X8}";`).
    *   This provides a blazing-fast, allocation-efficient plain text representation used by CLI loggers, debuggers, and clipboard actions.
2.  **`public IEnumerable<FormattedToken> GetFormatTokens()`**
    *   This is a duck-typed method (requiring no interface dependency on `CycloneDDS.Runtime`).
    *   The generator splits the template, yielding a distinct `FormattedToken` for every literal string segment and every evaluated field.
    *   The field's `.ToString("FormatString")` is automatically mapped to the specified `TokenType`.

**4. UI Integration (`DetailPanel`)**
When the `DetailPanel` encounters a custom struct or topic, it uses fast reflection to check if the `GetFormatTokens()` method exists on the target object type. If the method is found, the UI invokes it, iterates over the returned token sequence, and maps each `TokenType` to the corresponding CSS class (e.g., `detail-tree__value is-number`) to render a syntax-highlighted inline preview. If the method is absent, it falls back to the standard `ToString()` output.

**5. Example Usage**

*Schema Definition (Developer Input):*
```csharp
[DdsStruct]
[DdsTypeFormat("{Site:D:Number}:{App:D:Number}:{Entity:X8:Keyword}")]
public partial struct DisEntityId
{
    public ushort Site;
    public ushort App;
    public uint Entity;
}
```

*Generated C# Code (`CycloneDDS.CodeGen` Output):*
```csharp
// <auto-generated />
public partial struct DisEntityId
{
    // Native marshaling methods...

    public override string ToString() => $"{Site:D}:{App:D}:{Entity:X8}";

    public IEnumerable<CycloneDDS.Runtime.Formatting.FormattedToken> GetFormatTokens()
    {
        yield return new FormattedToken(this.Site.ToString("D"), TokenType.Number);
        yield return new FormattedToken(":", TokenType.Punctuation);
        yield return new FormattedToken(this.App.ToString("D"), TokenType.Number);
        yield return new FormattedToken(":", TokenType.Punctuation);
        yield return new FormattedToken(this.Entity.ToString("X8"), TokenType.Keyword);
    }
}
```
