## Context

The filtering engine currently allows users to evaluate fields but is lacking standard string-matching configurations and robust operator alternatives for command line execution. This batch resolves both problems by surfacing string filtering correctly and translating CLI-safe textual operators before dynamic compilation.

**Related Tasks:**
- [ME1-T04](../docs/mon-ext-1/ME1-TASK-DETAILS.md#me1-t04--startswith--endswith-in-filter-builder-ui) - String Methods implementation
- [ME1-T05](../docs/mon-ext-1/ME1-TASK-DETAILS.md#me1-t05--cli-safe-filter-operators) - CLI safe substitution

---

## 🎯 Batch Objectives

- Fully implement `.StartsWith()` / `.EndsWith()` / `.Contains()` in Linq expressions generated by the filter engine.
- Expose those strictly for string fields in the UI.
- Allow users to write `gt`, `lt`, `eq`, `ne`, `ge`, and `le` on the command line safely, converting them correctly before Linq compiles them.

## ⚠️ Common Pitfalls to Avoid
- **Greedy replacements**: A simplistic `.Replace(" ge ", " >= ")` is unsafe because operators might sit at the very start (`gt 50`) or end of a string without leading/trailing spaces.
- **LINQ Translation vs Raw Strings**: Do not mix variable interpolation indiscriminately when binding variables to the LINQ expression parameters in `FilterNodes.cs`. Use strongly typed parameters where expected (`@0`).
