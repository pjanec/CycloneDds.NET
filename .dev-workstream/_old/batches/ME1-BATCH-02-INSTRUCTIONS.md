### Task 0: Corrective Fix — 8-bit Enum Union Discriminators

**Files:**
- `tools/CycloneDDS.CodeGen/IdlEmitter.cs`

**Description:**
A bug was identified from the previous batch where `IdlEmitter.EmitUnion()` outputs integer literals (e.g., `0`, `2`) instead of string enumerator names for `[DdsUnion]` discriminators if the underlying enum is an 8-bit or 16-bit mapped type (like `StatusLevel : byte`). Because `val is int iVal` fails to match narrower primitives, the lookup fails and defaults to the integer literal. 
CycloneDDS `idlc` rejects integer literals for enum-switch unions (`Cannot evaluate integer literal as enumerator expression`).

**Detailed Specs & Success Conditions:**
- Ensure the union generator correctly casts/converts `val` to an integer uniformly so that the `enumDef.TypeInfo.EnumMembers` mapping works for all typed enums.

**Tests Required:**
- ✅ The newly failing test in `SelfSendTopics.cs` within `DdsMonitor.Engine.Tests` must compile and verify cleanly.

## ⚠️ Common Pitfalls to Avoid
- **Greedy replacements**: A simplistic `.Replace(" ge ", " >= ")` is unsafe because operators might sit at the very start (`gt 50`) or end of a string without leading/trailing spaces.
- **LINQ Translation vs Raw Strings**: Do not mix variable interpolation indiscriminately when binding variables to the LINQ expression parameters in `FilterNodes.cs`. Use strongly typed parameters where expected (`@0`).
